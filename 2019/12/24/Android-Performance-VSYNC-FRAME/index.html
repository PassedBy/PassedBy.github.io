2015新年伊始，Google发布了关于Android性能优化典范的专题，一共16个短视频，每个3-5分钟，帮助开发者创建更快更优秀的Android App。课程专题不仅仅介绍了Android系统中有关性能问题的底层工作原理，同时也介绍了如何通过工具来找出性能问题以及提升性能的建议。主要从三个方面展开，Android的渲染机制，内存与GC，电量优化。下面是对这些问题和建议的总结梳理。

0. Render Performance

大多数用户感知到的卡顿等性能问题的直接根源都是因为渲染性能。从设计师的角度来说，他们希望App能有更多的动画，图片等时尚元素来实现丰富而”流畅”的用户体验，但是Android系统很可能无法及时完成那些复杂的界面渲染操作。Android每隔16ms发出VSYNC信号，触发对UI的渲染，如果每次都渲染成功，这样就能够达到流畅的画面所需的60fps，反过来，为了实现60fps，这意味着程序的大多数操作必须在16ms内完成。

理想的情况是每隔16ms都能计算出相应的帧，以提供给系统进行渲染，然而并不是所有的情况都是理想的，还有很多情况不理想。

用户容易在UI执行动画或者滑动ListView的时候感到卡顿和不流畅，是因为这里的操作相对复杂，容易发生丢帧的现象。有很多原因可能导致丢帧，有可能是layout过于复杂，无法在16ms内完成渲染，有可能是因为你的UI上有层叠太多的绘制单元(Overdraw)，还有可能是动画执行的次数过多（动画往往涉及到大量计算）

发现并定位问题：
比如可以使用 HierarchyViewer来观察Activity中的布局是否过于复杂，也可以使用手机设置里提供的开发者工具：打开Show GPU Overdraw等选项进行观察；你还可以使用TraceView来观察CPU的执行情况，更加快捷地找到性能瓶颈。

1. Understanding Overdraw

Overdraw描述的是屏幕上的某个像素在同一帧的时间内被绘制了多次。在多层次的UI结构里面，如果不可见的UI也在做绘制的操作，这就会导致某些像素区域被绘制了多次。这就浪费了大量的CPU以及GPU资源。

幸运的是，我们可以通过手机设置里面的开发者选项打开Show GPU Overdraw，可以观察UI上的Overdraw情况

容易忽略的原因是，非必须的重叠背景，例如某个Activity有一个背景，然后里面的Layout区域又有自己的背景，同时子View又分别有自己的背景。例如子View设置”背景”的几种不同方法有什么区别，对性能有什么影响，都需要理清楚。

2. Understanding VSYNC

为了理解App是如何进行渲染的，我们必须了解手机硬件是如何工作的，那么就必须理解什么是VSYNC。在讲解VSYNC之前，我们需要先了解两个相关的概念：

  • RefreshRate: 代表了屏幕一秒内刷新屏幕的次数，这取决于硬件的固定参数，例如60Hz
  • Frame Rate: 代表了GPU在一秒内绘制操作的帧数，例如30fps，60fps

对于上面所讲，有点疑惑的地方是，VSYNC时间间隔和 RefreshRate之间的关系，因为前者一般都说的固定的，后者又说取决于硬件参数，还是可能VSYNC就是取决于RefreshRate，它也是可变的，只是通常是间隔16ms，也就是RefreshRate=60Hz的时候。

VSYNC 和 Tripple Buffer
要了解一个东西，往往要知道它的来龙去脉，最初的时候，并没有这些。Vsync其实就是一个脉冲信号，和屏幕的刷新频率直接关联。Vsync又不仅仅包含这一层意思，它还包含另一层意思，就是垂直同步，这一层意思是在慢慢发展的过程中获得的。
我们先来看最初屏幕是显示图像的。

单Buffer


首先要注意一点，屏幕从上到下刷新的速度是极快的，这个时间相对于两次脉冲信号，也就是屏幕再次刷新的间隔是可以忽略不计的。这也是为了便于大家对Tearing现象的理解。

假设屏幕刚刷新完第n帧数据，此时缓冲区A和屏幕都是第n帧数据。那么接下来GPU要准备第n+1帧数据了，很快它也准备完毕了，可是此时显示器还没有接收到刷新信号，可是GPU继续刷新着第 n+2 帧数据，此时刚好往缓冲区填了一半，此时缓冲区的内容为上半部分为 n+2,下半部分为 n+1 。不凑巧的是，display此时接到了下一个脉冲信号，它要开始刷新 “第n+1” 帧了，可是从上到下扫描完了以后，发现屏幕上同时显示了两帧的数据，画面出现 “撕裂” 现象，也就是 Tearing 现象。如下图所示：

有了问题，我们就要去解决，那么如何解决这个问题呢？我们首先想到了同步处理：
当第N+1帧数据没有被dispaly显示的时候，不让GPU继续往缓冲区写入数据，看起来是个不错的方法，但是却大大牺牲了性能,于是我们引入了双重缓存。

双重缓存

两个缓存区分别为 Frame Buffer 和 Back Buffer，GPU向Back Buffer中写数据，屏幕从FrameBuffer读取数据，Vsync信号负责二者的交换，然后通知CPU和GPU绘制下一帧图像，这样，当帧率大于刷新平吕时，帧率会被迫跟刷新频率保持相同，同时避免 “Tearing” 现象。
不过需要注意的是，当Vsync信号产生时，如果此时 GPU/CPU 正在生产帧数据，此时不会发生 Swap，屏幕也不会从 FrameBuffer中读取数据，即两个刷新周期显示的是同一帧数据，这就引来了新的问题，掉帧，在我们视觉上看来就是卡顿。可以来看一张图：

如何解决这个问题又摆在面前，可以看到，当jank发生的时候，大量的CPU/GPU被浪费，由于在Vsync到来时，它们还在工作，所以不能开始下一帧的绘制（也很明显，这一帧都没搞完）当然，我们能也是通过这种等待来避免Tearing发生的。但是，有没有可能进行改进呢。答案就是三重缓存。

三重缓存

这个地方有点绕，请先明确一点，也是我们之前讲到的一句话：

在某个Vsync来临之时，如果CPU/GPU仍在进行着绘制的操作，则不能继续计算下一个帧

假设这是某个Vsync到来时的当前状态：

Display: N
FrameBuffer: N
BackBuffer_1 : N+1 （还未渲染完毕）

实际上，往细了想，因为此时Back Buffer还没有写完，如果交换到了 Frame Buffer，则display是显示不完整的。不能计算下一个帧，是因为我们不能让此时的残缺的Back Buffer就交换到FrameBuffer，那么我们可不可以引入一个新的 Buffer_2呢，用它来绘制第N+2帧的数据（接管了原本下一次Buffer_1的任务），这样就相当于有两个 Back Buffer了，在不需要Swap的情况下（Framebuffer和display不用接收破东西了）开辟一个新的Buffer_2来绘制下一帧 N+2 的数据. 等到N+1帧的数据渲染完毕，此时交换FrameBufefr和BackBuffer_1 ,等下一个周期，如果 Buffer_2已经渲染完毕，那么再将 Buffer_2 Swap 到前面去显示。这样就解决了掉帧的问题，虽然在第一帧的时候还是掉了，但是后面是流畅的。

C接管了原来A应该绘制的帧 , 所以A后来渲染的其实是原本的A后面的一帧

5.SurfaceFlinger-图形合成者

SurfaceFlinger 接受来自多个来源的数据缓冲区，对它们进行合成，然后发送到显示设备。

当应用进入前台时，WindowManager 服务会向 SurfaceFlinger 请求一个绘图 Surface。SurfaceFlinger 会创建一个其主要组件为 BufferQueue 的层，而 SurfaceFlinger 是其消耗方。生产方端的 Binder 对象通过 WindowManager 传递到应用，然后应用可以开始直接将帧发送到 SurfaceFlinger。
